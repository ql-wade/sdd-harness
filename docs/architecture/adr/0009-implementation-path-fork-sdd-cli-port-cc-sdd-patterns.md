# Implementation path: fork sdd-ql-workflow and evolve, port cc-sdd patterns

SDD Harness fork 自 sdd-ql-workflow（ql-wade/sdd-cli 的 npm 包），在其基础上演进，而非作为独立 repo 依赖它。fork 保留 sdd-cli 已实现的约 70% 核心：`openspec/changes/<change-id>/` 目录、tracking 文件（`task_plan.md` / `findings.md` / `progress.md` / `delta-log.md`）、Trinity 的 new/continue/apply/verify/archive 流程、config.yaml/schema.yaml 配置体系，全部基于与 ADR-0001 一致的 OpenSpec 栈。在其上叠加 SDD Harness 层：9 个 sdd-* stage skill（封装 Trinity）+ `sdd-harness` 共享 skill + hooks + MCP + templates + governance。选择 fork 而非依赖，是因为 SDD Harness 对 sdd-cli 做深度改造（5 stage 扩 9 stage、加治理与知识闭环），走依赖会受 sdd-cli 接口约束；而 sdd-cli 体量小（~5 stars、更新慢），fork 的上游同步负担低。npm 包重命名为自有 scope（如 `@yourorg/sdd-harness`）。同时移植 cc-sdd 五项 workflow 模式：per-task independent reviewer、auto-debug（reviewer 拒 2 次后查根因）、boundary discipline（design.md 含 File Structure Plan，task 带 `_Boundary:_` / `_Depends:_` 注解）、learnings 经 findings.md 跨 task 传播、TDD behind feature flag。
